Ready-to-run jobs for AI agents.

A loadout is a ready-to-run job for an AI agent: the playbook, tools, guardrails, examples, and success criteria packaged into one portable markdown file. MCP gives agents tools — a loadout tells them how to use those tools to win the job. One file. Any MCP client.

Workflows do steps. Loadouts do jobs.

Read the format ↓ Validate a file GitHub

Tools are the floor

The market has settled the easy question — AI can control your apps through MCP. The hard question is whether it knows how to run the job well: a workflow says "when X, do Y"; a loadout says "here's how to win this whole outcome."

Tools · steps

40 tools, zero context. The agent guesses your audience, invents your offer, sends before you've seen anything, and calls it done because it stopped.

A loadout · the job

Interviews you first. Works the playbook. Refuses what the file forbids. Slams to a stop at the approval gate. Proves the job is done with a verifiable check.

Three non-negotiables

Everything else in the format bends. These don't.

1

Ask first

The file opens with a config interview — and the agent only asks what it can't discover from the file, the conversation, its memory, or the account itself. In a chat, these are questions; in a dashboard, the same lines render as a form. One artifact, two front-ends.

2

One approval gate, minimum

Before anything irreversible — a send, a post, an automation — the workflow stops and a human approves. A loadout without a gate is a script, not a loadout.

3

Done is verifiable

“Done means” must be checkable — name the read tool that proves the work happened. No vibes-based completion.

The format

Seven parts, in order. Full section reference in the spec README.

# <Name> — Loadout

```yaml
loadout: <slug>          # required — stable id
mcp: <server url>       # required — the MCP this file drives
tier: <access tier>    # optional
cadence: weekly        # optional — rhythm if run as a recurring loop
```

**The job:** one paragraph. The outcome, in the operator's language.

## Before you run — ask the user
Only ask what you don't already know. Check the Config, the conversation,
memory, and the account first. Confirm inferences in one line.
1. **Who is this for?** (hint)
2. **What's the asset?** (name + URL)

## Config
```yaml
audience:              # keys become stable form-field ids
asset:
```

## Steps
### 1. See — call `a_real_tool`, study the signal
### 2. Decide — rank, filter, dedupe against history
### 3. STOP — approval gate ⛔
Present everything. Nothing irreversible happens past here without a human.
### 4. Act — only what was approved · ### 5. Close the loop for next run

## Never
- The hard boundaries. No fabrication. No acting without approval.

## Done means
- Verifiable — name the read tool that proves it (`list_actions`, …)

Running one

    
Connect the MCP named in the file's header, then paste the whole file into your agent with this:

    
Load this Loadout and run it.

Follow the file exactly:
- ask me the config questions BEFORE doing anything else
- do not perform any irreversible action without my explicit approval
- when done, verify the result the way the file says to.

    
Works in Claude, ChatGPT, Cursor, or any MCP client. Dashboard runtimes render the same file as a form + checklist + approval queue — Apex by LeadShark is the reference runtime the format was proven against.
